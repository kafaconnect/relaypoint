Runbook — NATS single-node deploy

Bring up and verify the RelayPoint Phase-1 single-node signaling plane — NATS (JetStream durable + nats.ws + $SYS) + the router service + coturn — with one command.

Overview

The deployment lives entirely under deploy/: docker-compose.yml (the plane), nats/nats-server.conf (NATS + account ACLs), router.Dockerfile (the router image), .env.example (credentials), and verify.sh (the smoke check). It is a single NATS node and a single router instance — no HA. Media never transits NATS; coturn handles WebRTC NAT traversal.

Prerequisites

Docker with Compose v2 (docker compose). For verify.sh: curl or wget (one is enough). coturn uses host networking, so it needs the host's UDP range 49160–49200 free.

Bring up

cd deploy
cp .env.example .env        # optional for dev — the compose has matching :- defaults
docker compose up -d --build
docker compose ps

The router is gated on a NATS health check (depends_on: condition: service_healthy), so it connects only once NATS reports healthy on /healthz — a cold up does not race NATS startup.

Verify

./verify.sh                 # or: MON_URL=http://host:8222 ./verify.sh

It checks, via the NATS monitoring port, that NATS is healthy (/healthz), JetStream is enabled (/jsz), the INTERACTION_LOGS stream exists, and the router is connected (/connz). It exits non-zero on the first failure, so it works as a CI / post-deploy gate.

Ports & credentials

PortPurpose
4222NATS client (router, server SDKs)
8222Monitoring (/healthz, /jsz, /connz) — used by verify.sh
8088WebSocket (nats.ws browser clients); TLS terminated upstream in dev
1883MQTT — configured for the mobile phase, currently unused
3478 / 49160–49200coturn TURN listener / UDP relay range (host network)

Credentials come from .env (see .env.example); the compose ${VAR:-default} fallbacks keep a bare up working. The same ROUTER_PASSWORD authenticates the router NATS user (in nats-server.conf, env-expanded) and the router service — change it in one place.

Do not commit .env. It is gitignored. The values in .env.example are dev-only defaults; rotate them for anything beyond a local box.

Persistence

JetStream stores the durable interaction.<id>.log facts in the named volume nats-data (mounted at /data/jetstream). The log survives a container restart; docker compose down -v deletes it.

Security boundary

The NATS account ACLs are the edge backstop for the router-authoritative model: the client user is denied publish on tenant.*.interaction.*.log and may only publish .cmd / .signal.*; the router user is the sole writer of .log. So a client cannot forge a fact even if the router has a bug. $SYS is the system account.

Phase-1 caveat — not production authorship security. Because Phase-1 runs a shared client NATS user (no per-connection identity yet), a client can forge another user's actor_id on .cmd, publish another user's .signal.*, and subscribe another issuer's _INBOX.>. The ACL still prevents direct .log writes. Verified authorship + own-author-only signals + inbox isolation require the deferred auth-callout (NKEY/JWT) that mints a per-connection identity — a blocking precondition for production.

Teardown

docker compose down          # stop, keep the JetStream volume
docker compose down -v       # stop and DELETE the durable log

Deferred (later phases)

The 3-node JetStream RAFT HA cluster + router HA, NKEY/JWT auth + auth-callout, TLS at the NATS edge, and Kubernetes/k3s manifests are out of scope for this single-node Phase-1 deploy.
