Containers

C4 Level 2 view of RelayPoint: the deployable containers — a single-node NATS server, the authoritative router/interaction service, the presence service, coturn, the observability stack, and the web client — and the protocol on every relationship.

#Overview

At the container level RelayPoint is a small set of cooperating processes around one NATS server. In Phase 1 the NATS server runs as a single node with JetStream enabled and three listeners active — WebSocket, MQTT, and the system account that exports $SYS events. The router / interaction service is the authoritative core: it is the ONLY writer of every interaction.<id>.log fact and owns the offer, call, and interaction state machines. The presence service, coturn, and the observability stack are separate processes; the web client is browser-side code using nats.ws and the browser WebRTC stack.

A 3-node JetStream RAFT cluster is the HA path and is deliberately deferred. Everything below describes the single-node Phase 1 topology.

#Containers

#NATS server

One NATS server is the single backbone for both client signaling and microservices (NATS micro). It runs these cooperating facilities in one process:

  • websocket{} listener — accepts nats.ws browser connections over WSS.
  • mqtt{} listener (3.1.1) — configured for the future mobile bridge, idle in Phase 1. The a.b.c ↔ a/b/c auto-map keeps that bridge rework-free.
  • auth callout — authorization { auth_callout { … } } delegates per-connection authorization to the router so per-interaction subjects can be minted at CONNECT time from a short-lived, interaction-scoped token. See the subject model.
  • JetStream — file storage for the durable streams INTERACTION_LOGS, NOTIFICATIONS, and ROUTING_AUDIT; core NATS carries the ephemeral .cmd, .signal, presence, and live offer/control traffic.
  • KV bucket offer.active — a JetStream KV store, keyed offer.active.<userId> with TTLs, where the router persists active (ringing) offers so reconnecting clients can reconstruct them and a crash leaves no stuck ring.
  • $SYS account — exports $SYS.ACCOUNT.*.{CONNECT,DISCONNECT} connection events the presence service consumes.

#Router / interaction service

A stateful backend process (Go or TS, built on NATS micro) that is the authoritative core of RelayPoint. It is the single writer of every interaction.<id>.log fact: clients never write .log and never assign sequence. Clients publish intents as COMMANDS on interaction.<id>.cmd; the router subscribes .cmd, validates (tenant, actor role, state-machine legality, author == connection identity, payload tenant_id == subject tenant), assigns a monotonic per-interaction sequence, and appends the authoritative fact on .log.

The router owns:

  • The offer, call, and interaction state machines — see State Machines. Illegal transitions and forged authorship are rejected.
  • The per-interaction sequence counter (router-assigned, never client-set).
  • RONA and timers — offer timeout, fast-RONA on NATS 503 no responders, penalty-box backoff for agents who RONA/reject repeatedly.
  • The NATS-KV offer store (offer.active.<userId>) with TTLs, so reconnecting clients reconstruct ringing offers and a router crash leaves no stuck-ringing or phantom-assignment state (TTL sweeper + idempotent terminal transitions).
  • The orphaned reaper — when all participants are offline (per presence) for more than N minutes it injects interaction.ended with reason orphaned.
  • Auth-callout authorization — it answers the NATS auth callout, minting each connection's allowed subjects from a short-lived, nonce/audience-bound, actor+interaction-scoped token, and issues the interaction-scoped token on offer-accept. Privileged controls (cancel/withdraw/transfer/ACL grant-revoke) are audited on routing.audit.>.
Warning The router is the ONLY component with write access to .log. Clients are READ-only on .log and write-only on .cmd. This is what prevents authorship spoofing and sequence collisions; see the subject model.

#Presence service

A backend process (Go or TS) that subscribes to $SYS.ACCOUNT.*.{CONNECT,DISCONNECT} and publishes per-user state on presence.<userId>. Presence is derived from connection events, not from MQTT Last-Will-and-Testament.

It holds two NATS connections: it connects under the system (SYS) account to subscribe $SYS.ACCOUNT.*.{CONNECT,DISCONNECT} (only the SYS account can read $SYS), and publishes presence.<userId> on the APP account over a separate APP connection.

#coturn

A STUN/TURN server for WebRTC NAT traversal. Clients use STUN to discover candidates; coturn relays media via TURN only when a direct peer-to-peer path cannot be established. coturn is not a NATS client — it sits entirely on the media plane.

#Observability

nats-surveyor and prometheus-nats-exporter scrape NATS monitoring and $SYS endpoints and expose metrics to Prometheus; Grafana renders dashboards. All four dependencies are 100% open source, consistent with the project's hard rule.

#Web client

Browser-side code combining two stacks: nats.ws for the signaling channel over the websocket{} listener, and the browser WebRTC API (RTCPeerConnection) for media. The client follows strict subscription discipline — it subscribes to specific subjects and filters by event_type locally, never to the interaction.<id>.> wildcard. See the subject model for the rules.

#Media never touches NATS

Danger The NATS server carries only signaling — chat, presence, notifications, and WebRTC SDP/ICE. Audio and video MUST flow WebRTC peer-to-peer (relayed by coturn only when NAT requires it) and MUST NEVER transit the NATS server. A future SFU/media-server is a separate concern; it does not change this rule.

This boundary is what keeps the single-node NATS server viable: it never bears media bitrate, only low-rate control messages and the brief SDP/ICE exchange during call setup.
