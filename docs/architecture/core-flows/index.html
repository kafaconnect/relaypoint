Core Flows

The dynamic views of RelayPoint — how the subjects are used over time for offer/RONA, cancel/withdraw, double-accept, chat, calls, transfer, reconnect-during-ring, and abandon/orphan — plus the single-node Phase 1 deployment view. Throughout, clients send COMMANDS on .cmd and the router writes the authoritative .log.

#Overview

These flows show the subject model and state machines in motion. Each flow is a numbered sequence with an accompanying box-art diagram. Throughout, every subject is tenant-prefixed with tenant.<tenantId>. (omitted in the steps for brevity), and media never transits NATS.

Note Clients are READ-only on interaction.<id>.log and publish intents as COMMANDS on interaction.<id>.cmd. The router / interaction service validates each command, assigns sequence, and is the only writer of .log. Offer terminals are pushed by the router on routing.offer.user.<userId>.control.
Danger In the call flow, only SDP (on .log) and ICE (on .signal) cross NATS. The audio/video media itself flows WebRTC peer-to-peer, relayed by coturn only when NAT requires it — never through the broker.

#Offer to accept or RONA

The router brings a target user into an interaction via NATS request/reply with a timeout. See the offer state machine for all terminal states.

  1. The router publishes a request to routing.offer.user.<userId> with an offer_id, timeout_ms, and a one-time nonce, and persists the active offer in NATS KV offer.active.<userId> (TTL).
  2. The client's offer subscription receives the offer and shows the ring to the user.
  3. Accept path: the callee's client replies accept with the matching nonce on the request _INBOX before the timeout.
  4. The router drives the offer to accepted, issues the callee a short-lived interaction-scoped token, and returns the interaction_id. The callee reconnects with the token so the auth callout grants interaction.<id>.>. The router records interaction.assigned / participant.joined on interaction.<id>.log — authored by the router, NOT a caller→callee message.
  5. The client subscribes to interaction.<id>.log (and .signal once it joins media) and consumes the durable backlog.
  6. RONA path: if no valid accept arrives before the timeout (→ timed_out_rona), or the client replies reject (→ rejected), the router requeues the interaction; the user is not joined. If the target has no subscriber, NATS returns 503 no responders and the router fast-RONAs (→ no_responder_fast_rona) immediately.
  7. Every terminal is pushed by the router on routing.offer.user.<userId>.control so the client stops ringing.
 Router                       NATS                        Agent client
   |                            |                                |
   |-- persist offer.active.<userId> (KV, TTL) --               |
   |-- req: routing.offer.user.<userId> (offer_id, timeout, nonce) -->|
   |                            |        offer / ring            |
   |                            |                                |
   |   == ACCEPT (within timeout) ==                            |
   |<------ reply on _INBOX: accept (nonce) --------------------|
   |-- accepted; issue scoped token + return interaction_id --->|
   |                            |   reconnect w/ token; callout grants
   |                            |   sub interaction.<id>.log     |
   |-- write .log: interaction.assigned / participant.joined -->|
   |                            |<-- consume durable backlog ----|
   |                            |                                |
   |   == RONA (timeout / reject / 503 no responders) ==        |
   |<----------- reply: reject / (no reply) / 503 --------------|
   |-- requeue interaction (user NOT joined) --                 |
   |-- push terminal on routing.offer.user.<userId>.control --->|

#Offer cancel and withdraw

A ringing offer can end before the callee answers: the caller cancels (hangs up mid-ring) or the router withdraws (e.g. the customer abandoned). In both cases the router pushes the terminal on the target's .control subject so the callee stops ringing immediately.

  1. Cancel: the originating caller sends a cancel command on interaction.<id>.cmd. The router checks ACL + author (only the originator/router may cancel), drives the offer to cancelled, and clears the KV entry.
  2. Withdraw: on interaction.abandoned the router drives ringing offers to withdrawn.
  3. The router publishes { offer_id, terminal, reason } on routing.offer.user.<target>.control.
  4. The callee's .control subscription receives it and stops ringing — even though the callee never replied.
 Caller        NATS           Router                 Callee (ringing)
   |             |               |                         |
   |-- cmd: cancel (.cmd) ------>|                         |
   |             |   validate ACL + author (originator?)   |
   |             |               |-- offer -> cancelled    |
   |             |               |-- clear offer.active KV |
   |             |               |-- push .control: {offer_id, cancelled, reason} -->|
   |             |               |                         |  stop ringing
   |             |                                         |
   |   == WITHDRAW (router-initiated, e.g. abandoned) ==   |
   |             |               |-- offer -> withdrawn --->|
   |             |               |-- push .control: {offer_id, withdrawn, abandoned} -->|
   |             |                                         |  stop ringing

#Double-accept race

When an offer is fanned out to multiple agents (or devices), two accept replies can race. The router resolves it with a compare-and-set on offer_id / route_attempt_id: the first valid accept wins, the loser is told accepted_elsewhere.

  1. The router rings agents A and B for the same offer_id.
  2. Both reply accept on their own _INBOX nearly simultaneously.
  3. The router CAS-sets the winner (say A) → accepted; A is assigned, granted the interaction ACL, and joins.
  4. The later accept (B) loses the CAS → accepted_elsewhere / 409; the router pushes that terminal on B's .control and B stops ringing without joining.
  5. A repeated accept from the winning device A is idempotent — no second join.
 Agent A      Agent B        Router                  NATS
   |            |              |-- ring offer_id to A and B -->|
   |-- accept --|------------>|                              |
   |            |-- accept -->|                              |
   |            |              |-- CAS(offer_id): A wins      |
   |<-- accepted (assigned, ACL granted) --|                 |
   |            |<-- accepted_elsewhere / 409 --|             |
   |            |   push B .control: {offer_id, accepted_elsewhere} -> B stops ringing
   |-- accept (retry, same device) -->| idempotent: no second join

#Sending a chat message

A chat message is a durable fact, so the sender sends a command and the router writes it on .log, fanning out to all subscribed participants; offline participants receive it on reconnect.

  1. The sender publishes a send_message command with medium: chat on interaction.<id>.cmd. The sender does NOT set sequence and cannot write .log.
  2. The router validates (tenant, role, state, author == connection identity), assigns the next sequence, and appends one message.created fact on interaction.<id>.log.
  3. JetStream stores the event in INTERACTION_LOGS (with Nats-Msg-Id = event_id for dedup), preserving per-interaction order via sequence.
  4. Online participants subscribed to interaction.<id>.log receive it and filter by event_type locally.
  5. An offline participant, on reconnect, consumes the stream and receives the message in order (durable, not lost).
  6. In Desk, the message.created fact lands on the session's CustomerTimeline.
 Sender         NATS         Router        JetStream        Participants
   |             |             |               |                 |
   |-- cmd: send_message (.cmd) -->|           |                 |
   |   {medium: chat}  (no sequence)           |                 |
   |             |   validate; assign sequence=N                 |
   |             |             |-- append .log: message.created (seq N) -->|
   |             |             |   store in INTERACTION_LOGS     |
   |<----------- .log fanout to online subs -------------------- |
   |             |                                               |
   |             |   (offline participant reconnects)            |
   |             |<-- consume stream from cursor (in order) ---- |
Note Typing indicators for the same chat go on interaction.<id>.signal (ephemeral) with event_type: typing.* — they are never stored and never enter the timeline.

#Placing a call

A call escalates an existing interaction; it reuses the same interaction subjects. SDP offer/answer are durable state and go on .log; ICE candidates are ephemeral and go on .signal; the actual media is WebRTC peer-to-peer.

Precondition The offer is already accepted and the callee is in the interaction (see Offer to accept or RONA). The offer's terminal is accepted (on the offer reply / control); the resulting facts the router writes on .log are interaction.assigned / participant.joined — neither is part of the WebRTC setup below.
  1. The caller sends a webrtc.offer command (SDP) on interaction.<id>.cmd; the router records it as webrtc.offer on interaction.<id>.log — durable so a reconnecting peer can rebuild the call. See the call state machine.
  2. The callee sends a webrtc.answer command (SDP) on .cmd; the router records webrtc.answer on .log — also durable. The call advances setup_offered → answered.
  3. On joining the call, each peer subscribes to ICE and exchanges webrtc.ice candidates both ways on interaction.<id>.signal (ephemeral, rate-limited). Incoming ICE is buffered until the matching SDP from .log is applied. Candidates are discovered via coturn STUN. The call advances ice_connecting → connected.
  4. The peers establish a direct WebRTC connection and audio/video flows peer-to-peer. If NAT blocks a direct path, coturn relays the media via TURN.
  5. No media packet passes through NATS — only SDP (recorded on .log) and ICE (on .signal) did, during setup.
 Caller            NATS        Router       Callee          coturn
   |                |             |             |               |
   |   (precondition: offer accepted; callee in interaction)   |
   |-- cmd: webrtc.offer (SDP) (.cmd) -->|       |             |
   |                |   record .log: webrtc.offer (durable) -->|
   |                |             |<- cmd: webrtc.answer (.cmd)-|
   |<-- record .log: webrtc.answer (durable) ----|             |
   |   (join call -> subscribe ICE on .signal; buffer until SDP applied)
   |-- webrtc.ice (.signal, ephemeral) -------->|              |
   |<- webrtc.ice (.signal, ephemeral) ---------|              |
   |   STUN candidate discovery --------------+-------------->|
   |                |             |             |               |
   |============ WebRTC P2P media (A/V) ========|              |
   |   direct path, or TURN relay via coturn if NAT blocks direct
   |   *** media NEVER transits NATS ***       |              |

This is the unified-Interaction model at work: the same interaction.<id> subjects carried the earlier chat and now carry the call, differing only by event_type and medium in the payload.

#Transfer (warm and cold)

An agent transfers an active interaction to another target. The router offers the new target, bridges the legs on accept, and swaps the ACLs. Warm transfer keeps a brief overlap (both agents present); cold transfer has no overlap (the first agent leaves before the second joins).

  1. Agent A sends an interaction.transfer.requested command on interaction.<id>.cmd naming the new target B.
  2. The router rings B via routing.offer.user.<B> (the offer state machine). On accept, B reconnects with an interaction-scoped token and the auth callout grants B interaction.<id>.>.
  3. The router records interaction.transferred on .log, grants the new leg's ACL, and revokes the old leg's ACL.
  4. Warm: A and B overlap briefly (A's ACL revoked after B is connected). Cold: A leaves first (participant.left), then B joins — no overlap.
 Agent A        NATS        Router            Agent B (new target)
   |             |             |                     |
   |-- cmd: interaction.transfer.requested (.cmd) ->|
   |             |   ring routing.offer.user.<B> --->|
   |             |             |<-- accept (nonce) --|
   |             |             |   grant B ACL (reconnect w/ token)
   |             |             |-- .log: interaction.transferred -->|
   |             |             |   warm: A + B overlap, then revoke A
   |             |             |   cold: revoke A (participant.left) before B joins
   |<-- ACL revoked ---------- |                     |<-- ACL granted

#Reconnect during ring

A ringing client can drop and reconnect before answering. Because the router persists active offers in NATS KV offer.active.<userId>, the client reconstructs its ringing offers and resumes — no offer is lost.

  1. The router rings the client and persists the offer in KV offer.active.<userId> with a TTL.
  2. The client's connection drops mid-ring (no reply sent).
  3. The client reconnects (transparent over nats.ws) and, via the router, reads offer.active.<userId> from KV to reconstruct its ringing offers; it resumes ringing.
  4. A client-local ring timeout is the backstop if no .control message arrives. If the router crashed, a TTL sweeper plus idempotent terminal transitions ensure no stuck ring and no phantom assignment.
 Router            NATS (KV)              Client
   |                  |                     |
   |-- ring + persist offer.active.<userId> (TTL) -->|
   |                  |        ring         |
   |                  |                     X  connection drops (no reply)
   |                  |                     |
   |                  |        (reconnect)  |
   |                  |<-- read offer.active.<userId> (reconstruct) --|
   |                  |        resume ring  |
   |   backstop: client-local ring timeout if no .control arrives    |
   |   crash: TTL sweeper + idempotent terminals -> no stuck ring     |

#Customer abandoned and orphaned reaper

Two related lifecycle endings: the customer abandons (leaves before or after assignment), and the orphaned reaper ends an interaction whose every participant has been offline too long. See the interaction state machine.

  1. Abandon: the customer leaves; the router records interaction.abandoned and withdraws any ringing offers (terminal withdrawn, pushed on the target's .control), so a ringing agent stops ringing.
  2. Offline vs left: a transient drop records participant.offline (presence-driven, recoverable) and does NOT remove the participant; an explicit leave records participant.left (permanent).
  3. Reaper: when every participant has been offline (per presence) for more than N minutes, the reaper injects interaction.ended with reason orphaned.
 Customer        Router            Presence          Ringing agent
   |               |                  |                   |
   |  leaves ----> |                  |                   |
   |               |-- .log: interaction.abandoned        |
   |               |-- withdraw offers -> .control: {withdrawn} -->| stop ringing
   |               |                  |                   |
   |   == later: orphaned reaper ==   |                   |
   |               |<- all participants offline > N min --|
   |               |-- .log: interaction.ended [orphaned] |
   |               |   (participant.offline = transient; participant.left = permanent)

#Deployment view

Phase 1 deploys as a single-node topology via docker-compose: one NATS server (JetStream + WebSocket + MQTT + $SYS), coturn, the presence service, and the observability stack. The HA cluster is deferred. 

 docker-compose (single host)
 +-------------------------------------------------------------+
 |  nats         single-node NATS                              |
 |               JetStream (file) + websocket{} + mqtt{} + $SYS|
 |  coturn       STUN / TURN                                   |
 |  presence     $SYS -> presence.<userId> (Go/TS)             |
 |  observability surveyor + nats-exporter -> Prometheus       |
 |                -> Grafana                                   |
 +-------------------------------------------------------------+
   Browsers connect over WSS (nats.ws) + STUN/TURN to coturn.
   Media is WebRTC P2P between browsers (TURN relay only if NAT).
Warning This is a single node — no high availability. A 3-node JetStream RAFT cluster is the HA path and is deliberately deferred to a later phase, along with the MQTT-mobile bridge, NKEY/JWT auth, and a server-side SFU/media-server.
