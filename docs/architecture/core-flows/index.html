Core Flows

The dynamic views of RelayPoint — how the subjects are used over time for offer/RONA, chat, and calls — plus the single-node Phase 1 deployment view.

#Overview

These flows show the subject model in motion. Each flow is a numbered sequence with an accompanying box-art diagram. Throughout, every subject is tenant-prefixed with tenant.<tenantId>. (omitted in the steps for brevity), and media never transits NATS.

Danger In the call flow, only SDP (on .log) and ICE (on .signal) cross NATS. The audio/video media itself flows WebRTC peer-to-peer, relayed by coturn only when NAT requires it — never through the broker.

#Offer to accept or RONA

The router brings a target user into an interaction via NATS request/reply with a timeout.

  1. The router publishes a request to routing.offer.user.<userId> with a timeout_ms.
  2. The client's offer subscription receives the offer and shows the ring to the user.
  3. Accept path: the client replies accept before the timeout.
  4. The router grants the user ACL on interaction.<id>.* and returns the interaction_id in the reply.
  5. The client subscribes to interaction.<id>.log (and .signal once it joins media) and consumes the durable backlog.
  6. RONA path: if no valid accept arrives before the timeout, or the client replies reject, the router requeues the interaction (redirect-on-no-answer); the user is not joined.
 Router                       NATS                        Agent client
   |                            |                                |
   |-- req: routing.offer.user.<userId> (timeout_ms) ---------->|
   |                            |        offer / ring            |
   |                            |                                |
   |   == ACCEPT (within timeout) ==                            |
   |<------------- reply: accept --------------------------------|
   |-- grant ACL interaction.<id>.*  + return interaction_id -->|
   |                            |   sub interaction.<id>.log     |
   |                            |<-- consume durable backlog ----|
   |                            |                                |
   |   == RONA (timeout or reject) ==                           |
   |<----------- reply: reject / (no reply) --------------------|
   |-- requeue interaction (user NOT joined) --                 |

#Sending a chat message

A chat message is a durable fact, so it goes on .log and fans out to all subscribed participants; offline participants receive it on reconnect.

  1. The sender publishes an envelope with event_type: message.created and medium: chat to interaction.<id>.log.
  2. JetStream stores the event in INTERACTION_LOGS, preserving per-interaction order via sequence.
  3. Online participants subscribed to interaction.<id>.log receive it and filter by event_type locally.
  4. An offline participant, on reconnect, consumes the stream and receives the message in order (durable, not lost).
  5. In Desk, the message.created fact lands on the session's CustomerTimeline.
 Sender            NATS / JetStream            Participants (online + offline)
   |                     |                                 |
   |-- pub interaction.<id>.log ------------------------->|
   |   {message.created, medium: chat, sequence: N}      |
   |                     | store in INTERACTION_LOGS      |
   |                     |-- fanout to online subs ------>| (online: deliver now)
   |                     |                                 |
   |                     |   (offline participant reconnects)
   |                     |<-- consume stream from cursor --| (offline: delivered in order)
Note Typing indicators for the same chat go on interaction.<id>.signal (ephemeral) with event_type: typing.* — they are never stored and never enter the timeline.

#Placing a call

A call escalates an existing interaction; it reuses the same interaction subjects. SDP offer/answer are durable state and go on .log; ICE candidates are ephemeral and go on .signal; the actual media is WebRTC peer-to-peer.

  1. The caller offers the call: an envelope with event_type: call.accepted/webrtc.offer semantics begins on interaction.<id>.log (the offer/ring to a not-yet-joined user uses the routing offer instead).
  2. The callee accepts; call.accepted is published on interaction.<id>.log (durable).
  3. The caller publishes webrtc.offer (SDP) on interaction.<id>.log; the callee replies with webrtc.answer (SDP) on .log — both durable so a reconnecting peer can rebuild the call.
  4. On joining the call, each peer subscribes to ICE and exchanges webrtc.ice candidates on interaction.<id>.signal (ephemeral). Candidates are discovered via coturn STUN.
  5. The peers establish a direct WebRTC connection and audio/video flows peer-to-peer. If NAT blocks a direct path, coturn relays the media via TURN.
  6. No media packet passes through NATS — only SDP (on .log) and ICE (on .signal) did, during setup.
 Caller                 NATS                  Callee            coturn
   |                      |                       |                |
   |-- call.accepted -----| (.log, durable) ----->|                |
   |-- webrtc.offer (SDP)-| (.log, durable) ----->|                |
   |<- webrtc.answer (SDP)| (.log, durable) ------|                |
   |   (join call -> subscribe ICE on .signal)    |                |
   |-- webrtc.ice --------| (.signal, ephemeral)->|                |
   |<- webrtc.ice --------| (.signal, ephemeral)--|                |
   |   STUN candidate discovery  -----------------+--------------->|
   |                      |                       |                |
   |============ WebRTC P2P media (A/V) ==========|                |
   |   direct path, or TURN relay via coturn if NAT blocks direct  |
   |   *** media NEVER transits NATS ***          |                |

This is the unified-Interaction model at work: the same interaction.<id> subjects carried the earlier chat and now carry the call, differing only by event_type and medium in the payload.

#Deployment view

Phase 1 deploys as a single-node topology via docker-compose: one NATS server (JetStream + WebSocket + MQTT + $SYS), coturn, the presence service, and the observability stack. The HA cluster is deferred. 

 docker-compose (single host)
 +-------------------------------------------------------------+
 |  nats         single-node NATS                              |
 |               JetStream (file) + websocket{} + mqtt{} + $SYS|
 |  coturn       STUN / TURN                                   |
 |  presence     $SYS -> presence.<userId> (Go/TS)             |
 |  observability surveyor + nats-exporter -> Prometheus       |
 |                -> Grafana                                   |
 +-------------------------------------------------------------+
   Browsers connect over WSS (nats.ws) + STUN/TURN to coturn.
   Media is WebRTC P2P between browsers (TURN relay only if NAT).
Warning This is a single node — no high availability. A 3-node JetStream RAFT cluster is the HA path and is deliberately deferred to a later phase, along with the MQTT-mobile bridge, NKEY/JWT auth, and a server-side SFU/media-server.
