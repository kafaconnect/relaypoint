RelayPoint Architecture

The architecture hub for RelayPoint — the NATS web-signaling backbone that powers real-time chat, presence, voice/video signaling, and notifications for KafaConnect Desk.

#Overview

RelayPoint is a real-time signaling backbone for web apps. One NATS server carries four channels — chat, presence, voice/video signaling, and notification — and doubles as the microservices fabric. It is the signaling service for KafaConnect Desk (consumed there as a submodule), but is built as a standalone, reusable backbone — not Desk-specific.

This hub indexes the architecture documents in C4 order — system context, then containers, then the subject model and the dynamic flows. Each document is self-contained and reads in isolation. The authoritative requirements live in OpenSpec; these documents explain and diagram them, they do not replace them.

Note "Open Channel" is only the UI design name for the Desk surface; the project itself is RelayPoint, the signaling backbone.

#Architecture documents

DocumentC4 levelWhat it covers
System Context L1 Actors (agents, customers, the routing engine, presence) and external systems (browsers via nats.ws, coturn, MQTT-mobile [deferred], observability), and how RelayPoint serves Desk.
Containers L2 The deployable parts — NATS server, presence service, coturn, observability, web client — and the protocol on every relationship.
Subject Model — The key document. The router-authoritative, unified-Interaction subject layout — command plane vs log, the event envelope, the QoS split, JetStream streams, offer/RONA mechanics, the KV offer store, subscription discipline, the Desk mapping, and auth-callout ACLs.
State Machines — The three machines the router owns and enforces — offer, 1:1 call/WebRTC, and interaction — each as ASCII plus a transitions table.
Core Flows — The dynamic views — offer/accept/RONA, cancel/withdraw, double-accept, transfer, reconnect-during-ring, abandon/orphan, chat, and call — plus the single-node deployment view.

#Reading order

New readers should follow C4 levels top-down, then read the subject model, then the flows:

  1. System Context — who and what RelayPoint talks to.
  2. Containers — the moving parts and their wire protocols.
  3. Subject Model — the heart of the design; everything else references it.
  4. State Machines — the offer, call, and interaction lifecycles the router enforces.
  5. Core Flows — how the subjects are used over time, plus deployment.

#Decision records

Architecture Decision Records (ADRs) capture the rationale behind individual locked choices. They live under docs/architecture/decisions/ and will be linked here as they are authored.

Note No ADRs are published yet. The locked decisions are summarized in openspec/project.md and detailed in the subject model until the first ADR lands.

#Source of truth

These documents explain the design; they do not own it. The binding artifacts are:

  • openspec/project.md — purpose, tech stack, locked decisions, conventions, phases.
  • openspec/changes/signaling-core/design.md — the unified-Interaction model and QoS split.
  • openspec/changes/signaling-core/specs/signaling-core/spec.md — the requirements and their test scenarios.

When a document and an OpenSpec artifact disagree, the OpenSpec artifact wins.
