State Machines

The three state machines the router/interaction service owns and enforces: the offer (ring) lifecycle, the 1:1 call/WebRTC lifecycle, and the interaction lifecycle. Each is shown as ASCII plus a transitions table. Illegal transitions are rejected and write no .log fact.

#Overview

RelayPoint's router is the single authoritative owner of three state machines: the offer (the ring that brings a user into an interaction), the call/WebRTC media negotiation, and the interaction itself. Clients send COMMANDS on interaction.<id>.cmd and offer replies on the request _INBOX; the router validates each one against the legal transitions below, then appends the authoritative fact on interaction.<id>.log (or pushes an offer terminal on routing.offer.user.<userId>.control). An illegal transition (for example, resuming an interaction that has already ended) is rejected and writes no fact.

Phase 1 scope Calls are 1:1 only. Multi-party / conference is explicitly deferred. The call machine below describes a single caller/callee pair; the interaction machine carries the medium as a payload field, so a chat can escalate to a 1:1 call within the same interaction.

Throughout, every subject is tenant-prefixed with tenant.<tenantId>. (omitted for brevity). These machines are the binding contract behind the core flows; the flows show specific paths through them.

#Offer state machine

The offer brings a target user who is not yet in the interaction into it, via NATS request/reply on routing.offer.user.<userId> (reply on the request _INBOX with a one-time nonce). It progresses offered → ringing → terminal, where terminal is exactly one of eight states. The _INBOX carries a single reply — the terminal accept / reject; ringing is implicit once the request is published with no 503 No Responders, and there is NO separate ringing reply (it would consume the inbox). The non-reply terminals (cancel/withdraw/accepted_elsewhere, and any timeout/expiry) are pushed by the router on routing.offer.user.<userId>.control so the ringing client stops immediately — NOT via the reply.

Double-accept on a fan-out is resolved by a compare-and-set on offer_id / route_attempt_id: the first valid accept wins (accepted); every later accept is answered accepted_elsewhere / 409; a repeated accept from the same winning device is idempotent. The no_responder_fast_rona path fires immediately on NATS 503 no responders (target offline / never-subscribed) without waiting the full timeout_ms — distinct from timed_out_rona. Active offers are persisted in NATS KV offer.active.<userId> with TTLs, so a reconnecting client reconstructs its ringing offers and a router crash leaves no stuck ring.

#Call / WebRTC state machine (1:1)

The call machine governs WebRTC media negotiation for a single caller/callee pair. SDP offer/answer are durable and recorded on .log (webrtc.offer / webrtc.answer); ICE candidates are ephemeral on .signal (webrtc.ice) and the media itself is WebRTC peer-to-peer — never on NATS. It runs idle → setup_offered → answered → ice_connecting → connected, then loops through renegotiating / held / transferring / reconnecting back to connected, with terminals cancelled, ended, media_failed, and setup_failed.

 Glare: simultaneous webrtc.offer -> perfect negotiation.
 Deterministic polite/impolite role (caller=impolite, else lexical userId tie-break);

#Interaction state machine

The interaction is the unifying conversation across all media. It runs started → offered → assigned → active, loops through held / escalated / transferring back to active, and terminates in ended, abandoned, or orphaned. The router rejects invalid transitions (no resume after ended) and writes no fact for them. 

  [started]
     |  router rings a target (offer machine)
     v
  [offered] --- customer leaves before assignment ---> [abandoned]
     |  offer accepted
     v
  [assigned] --- ACL granted to the assigned agent
     |
     v
 +------------------------ [active] ------------------------+
 |        ^             ^                ^                   |
 |        |             |                |                   |
 |     [held]      [escalated]     [transferring]           |
 |   call.held/    medium added    interaction.transfer.    |
 |   resumed       (chat->voice)   requested -> new target  |
 |                                 (ACL grant new/revoke old)|
 +----------------------------------------------------------+
     |                    |                       |
  ended by party    all participants offline   customer leaves
     |               > N min (reaper)            after assignment
     v                    v                       v
  [ended]            [orphaned]               [abandoned]
                  interaction.ended[orphaned]  router withdraws
                                               ringing offers

 participant.offline  = transient (presence-driven, recoverable)
 participant.left     = explicit/permanent (removes from interaction)
Offline vs left, abandon, reaper participant.offline is a transient presence fact and does not remove a participant; participant.left is explicit and permanent. When the customer abandons (interaction.abandoned), the router withdraws any ringing offers (terminal withdrawn). The orphaned reaper injects interaction.ended with reason orphaned when every participant is offline for more than N minutes.
