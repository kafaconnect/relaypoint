System Context

C4 Level 1 view of RelayPoint: the people and external systems it interacts with, and how it acts as the signaling service for KafaConnect Desk.

#Overview

RelayPoint is the real-time signaling backbone for web apps and the signaling service for KafaConnect Desk. At the system-context level it sits between web clients in the browser and the supporting infrastructure for presence, NAT traversal, and observability. Web clients are either agents or customers; both connect from the browser over a WebSocket.

RelayPoint carries control-plane traffic only — chat, presence, WebRTC SDP/ICE exchange, and notifications. It deliberately does not carry media: audio and video flow peer-to-peer over WebRTC, never through RelayPoint. Desk consumes RelayPoint (as a submodule) to power its agent and customer real-time experience.

Danger Media never touches RelayPoint. Only SDP and ICE candidates transit the backbone during call setup; audio/video go WebRTC peer-to-peer, relayed by coturn only when NAT requires it. Any design that routes media through NATS is wrong.

#Scope boundary

RelayPoint owns the signaling/control plane and explicitly does not own the media plane.

  • Owns: chat transport, presence, WebRTC SDP/ICE exchange, notifications — all over NATS subjects — plus the presence service and auth at the NATS edge.
  • Does not own: the media plane. Audio/video never touch NATS; WebRTC media flows peer-to-peer, relayed by coturn only when NAT requires it. A server-side SFU/media-server is a later concern.
