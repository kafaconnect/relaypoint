Subject Model

The heart of RelayPoint: the router-authoritative, unified-Interaction NATS subject layout — the command plane vs the authoritative log, the event envelope, the durable-vs-ephemeral QoS split, the JetStream streams, offer/RONA mechanics, the KV offer store, client subscription discipline, the Desk mapping, and auth-callout multi-tenant ACLs.

#Overview

Every chat, voice, video, email, or social exchange is modeled as one Interaction. The medium is a field in the event payload, never part of the subject — there are no medium-specific subjects such as call.invite or chat.offer. A single conversation that starts as chat and escalates to a voice call keeps the same interaction subjects throughout; only the event_type and medium in the payload differ.

Per interaction there are three subjects, split by role and quality of service: a write-only command subject .cmd (core NATS) where clients publish intents, a durable, ordered log .log on JetStream that ONLY the router writes, and an ephemeral, per-publisher signal .signal.<userId> on core NATS for high-rate transient signals (the publisher's id is in the subject, so the NATS publish-ACL enforces authorship; subscribers read .signal.*). Clients are READ-only on .log; they never write a fact directly and never assign sequence. The offer/ring that brings a user into an interaction is a separate, target-addressed routing concern with its own subtree, terminal-control subject, and timeout/RONA semantics — never on the interaction update subjects. All subjects are tenant-prefixed with tenant.<tenantId>. (omitted below for brevity).

Note Subjects are dot-separated and lowercase. Ids (tenantId, interaction id, userId) are ULID/UUID with no dots or slashes, so they never break the subject hierarchy or the a.b.c ↔ a/b/c MQTT auto-map.

#Why the router is authoritative

A stateful router / interaction service is the single authoritative writer of every interaction.<id>.log fact and the owner of the offer, call, and interaction state machines, the sequence counter, RONA/timers, the NATS-KV offer store, the orphaned reaper, and auth-callout grant/revoke. Clients are READ-only on .log and publish intents as COMMANDS on interaction.<id>.cmd. If clients could write .log directly, two things break:

  • Authorship spoofing. A client could publish a fact claiming any actor_id — recording a message or an interaction.ended "from" another agent. With a write-only .cmd plane, the router validates author == the connection's authenticated identity and rejects forged authorship.
  • Sequence collisions. sequence is a single monotonic per-interaction counter. Multiple writers cannot assign it without races and gaps; only a single authoritative writer can guarantee a clean, ordered log. Clients never set sequence.

The router also enforces state-machine legality (no resume after ended), tenant scope (subject tenant must match the payload tenant_id), and actor role. Illegal transitions are rejected and write no .log fact. See the state machines for the legal transitions.

Danger Clients hold NO write ACL on .log. A client that tries to publish message.created or interaction.ended on tenant.<tid>.interaction.<id>.log is denied by NATS. The only path to record a fact is a command on interaction.<id>.cmd.

#Command to validate to append

Every fact on .log originates as a client command on .cmd (or as a router-internal event such as a timer firing or the reaper running). The router validates each command and, only if it passes, assigns the next sequence and appends one authoritative fact.

  1. The client publishes a command — e.g. send_message, webrtc.offer, hold, cancel — on interaction.<id>.cmd.
  2. The router validates: tenant scope (subject tenant == payload tenant_id), actor role, state-machine legality, and author == the connection's authenticated identity.
  3. On pass: the router assigns the next monotonic sequence and appends one fact (e.g. message.created) on interaction.<id>.log, authored by the router.
  4. On fail (forged author, illegal transition, tenant mismatch): the command is rejected and NO .log fact is written.
 Client                NATS              Router                JetStream
   |                     |                  |                       |
   |-- pub .cmd: send_message ------------>|                       |
   |                     |   (sub .cmd)     |                       |
   |                     |                  |-- validate tenant/role/state/author
   |                     |                  |     pass -> assign sequence=N
   |                     |                  |-- append .log: message.created (seq N) ->|
   |<----- .log fanout: message.created (seq N) ------------------ |
   |                     |                  |     fail -> reject, NO .log written     |

#Event envelope

Every message — a command on .cmd, a fact on .log, or a signal on .signal — carries the same envelope. The business meaning lives in event_type and the medium lives in medium, both inside the payload. On .log, sequence is router-assigned; clients never set it.

{
  "schema":      "...",          // schema id / version of `data`
  "event_type":  "message.created",
  "event_id":    "<ULID>",       // Nats-Msg-Id for JetStream dedup
  "sequence":    42,             // ROUTER-assigned, per-interaction ordering
  "occurred_at": "2026-06-08T12:00:00Z",
  "tenant_id":   "<tenantId>",   // MUST match the subject's tenant
  "actor_id":    "<userId>",     // MUST equal the connection's identity
  "medium":      "chat",         // chat | voice | video | email | social
  "ref_id":      "<event_id>",   // optional: target of update/delete
  "data":        { /* event-specific */ }
}
FieldMeaning
schemaSchema id/version of the data payload.
event_typeThe business event; clients filter on this locally.
event_idUnique id for the event. Published as Nats-Msg-Id so JetStream dedups within its window; clients dedup beyond it.
sequencePer-interaction ordering counter — router-assigned on .log; clients never set it. Consumers track the last applied sequence and replay from JetStream on a detected gap.
occurred_atEvent timestamp.
tenant_idThe tenant. The router rejects a command whose tenant_id does not match the subject's tenant even if the subject ACL passes.
actor_idThe acting user. On a command the router requires actor_id == the connection's authenticated identity (forged authorship rejected).
mediumThe channel — chat, voice, video, email, social. This is the unified-Interaction key: the medium is data, not a subject.
ref_idOptional. On message.updated / message.deleted it carries the target event_id — applied as redaction (edit) or tombstone (delete).
dataEvent-specific payload.

#QoS split: log vs signal

The split is by quality of service, not by business domain. Durable facts that must be ordered and replayable go on .log (JetStream). High-rate transient signals that must never be stored go on .signal (core NATS). The event_type distinguishes specific events within each subject.

#.log event types

Durable, ordered facts on interaction.<id>.log — all written by the router (JetStream):

  • interaction.started / interaction.assigned / interaction.transferred / interaction.abandoned / interaction.ended
  • message.created / message.updated / message.deleted (carry ref_id)
  • participant.joined / participant.offline / participant.left
  • webrtc.offer / webrtc.answer
  • webrtc.renegotiation.offer / webrtc.renegotiation.answer (carry negotiation_id / generation)
  • call.answered / call.connected / call.held / call.resumed / call.cancelled / call.ended / call.media_failed / call.setup_failed
  • recording.started / recording.stopped
Note participant.offline is a transient, presence-driven fact (the participant may return); participant.left is explicit and permanent and removes them from the interaction. WebRTC SDP offer/answer are durable state and live on .log — a reconnecting participant must replay them to rebuild the call. ICE candidates are not state and live on .signal.

#.signal event types

Ephemeral, at-most-once signals on the per-publisher interaction.<id>.signal.<userId> (core NATS, never JetStream; subscribers read interaction.<id>.signal.*):

  • webrtc.ice — ICE candidates
  • typing.* — typing indicators
  • presence.cursor — live cursor position
  • media.level — audio level meters

#JetStream streams

Streams are defined by leaf class, not per interaction. Three streams are required; nothing else is stored.

StreamSubjects capturedStorage / notes
INTERACTION_LOGStenant.*.interaction.*.logFile storage; preserves per-interaction ordering; Nats-Msg-Id = event_id dedup.
NOTIFICATIONStenant.*.notify.*Durable; survives reconnect.
ROUTING_AUDITtenant.*.routing.audit.>Privileged-control audit (actor + reason); NOT live offer delivery.
Danger There is NO JetStream stream for .cmd, for .signal, for presence.*, or for the live offer / control subjects (routing.offer.*, routing.offer.user.<id>.control). The command plane and signals are core NATS only. Capturing .signal in JetStream would persist ICE floods and typing noise and defeat the entire QoS split. The KV bucket offer.active.<userId> is a separate JetStream KV store, not one of these streams.

#Offer mechanics

An offer brings a user who is not yet in an interaction into it. The router publishes a NATS request/reply to routing.offer.user.<userId> (team and queue variants exist) carrying an offer_id, a timeout_ms, and a one-time nonce; the target client replies on the request's _INBOX with the matching nonce (bound to tenant_id/offer_id/target). Offer delivery never shares a subject or stream with interaction updates. The offer is a full state machine — see the offer state machine — with terminal states accepted, rejected, cancelled, withdrawn, accepted_elsewhere, timed_out_rona, expired, and no_responder_fast_rona.

  • Accept within timeout — the client replies accept with the nonce before the deadline; the router drives the offer to accepted, grants the user the interaction ACL (via a reconnect with an interaction-scoped token), and returns the interaction_id.
  • No answer or reject → RONA — if no valid accept arrives before the timeout (→ timed_out_rona), or the client replies reject (→ rejected), the router requeues the interaction and the user is not joined.

#Offer control and cancel/withdraw

Every terminal transition is pushed by the router to routing.offer.user.<userId>.control as { offer_id, terminal, reason }, which the ringing client subscribes to so it stops ringing immediately — even when the terminal was not the client's own reply.

  • Cancel (caller): the originating caller sends a cancel command; the router drives the offer to cancelled and pushes it on the target's .control, so the callee stops ringing.
  • Withdraw (router): when the customer abandons (interaction.abandoned), the router drives ringing offers to withdrawn and pushes them on .control.
  • Authorization: only the originator or the router may cancel/withdraw a ringing offer; a cancel/withdraw from any other actor is rejected by ACL + author check.

#Double-accept CAS

When an offer is fanned out to multiple devices or agents, two accept replies can race. The router performs a compare-and-set on the offer_id / route_attempt_id: the first valid accept wins (accepted), and every later accept is answered accepted_elsewhere / 409. A repeated accept from the same winning device is idempotent — no second join.

#Fast-RONA

If the target user has no subscriber on routing.offer.user.<userId> (offline or never-subscribed), the NATS request returns 503 no responders immediately. The router drives the offer to no_responder_fast_rona at once and requeues — distinct from a no-answer timeout, and without waiting the full timeout_ms.

#KV offer store and reconnect

The router persists active (ringing) offers in the NATS KV bucket offer.active.<userId> with TTLs, writing one entry per fanned-out user: a team or queue offer fans out to several users, and the router writes offer.active.<userId> for EACH of them — there is no team-level KV the client must read. When a ringing client drops and reconnects, it reconstructs only its own pending offers from offer.active.<self> (via the router) and resumes ringing; a client-local ring timeout is the backstop if no control message arrives. On accept, withdraw, or any other terminal, the router clears those per-user KV entries; on router crash, a TTL sweeper plus idempotent terminal transitions ensure no offer is left stuck-ringing and no interaction is phantom-assigned.

#Subscription discipline

Clients subscribe to specific subjects and filter by event_type locally. They MUST NOT use the interaction.<id>.> wildcard, which would flood the client with every ICE candidate and signal.

  • Subscribe to the exact subjects you need: interaction.<id>.log, interaction.<id>.signal.*, routing.offer.user.<userId>, routing.offer.user.<userId>.control, notify.<userId>, presence.<userId>. Publish only on interaction.<id>.cmd and your own interaction.<id>.signal.<self>.
  • Filter by event_type in the client, not with a subject wildcard.
  • The WebRTC subsystem subscribes to ICE (on .signal) only when the user joins the call — not before — so a chat-only participant never receives ICE traffic. It buffers incoming ICE until the matching SDP (from .log) is applied.
Warning Never subscribe to interaction.<id>.>. The wildcard collapses the QoS split at the client and re-introduces the ICE flood the .log/.signal separation exists to prevent.

#Rate limits

The per-publisher interaction.<id>.signal.<userId> is rate-limited per user/interaction, with a cap on ICE candidates per negotiation. Because the publisher's id is in the subject, the NATS publish-ACL — not a payload actor_id check — binds each signal to its author: a client may publish only to its own .signal.<self>, and there is no shared signal subject on which authorship could be forged. Excess is dropped or throttled and never reaches JetStream — .signal is never durable. This protects the single-node broker from ICE floods and typing storms while keeping the durable .log path clean.

Note Rate limiting applies to the ephemeral signal plane. The command plane (.cmd) is validated per-message by the router, which is itself the natural backpressure point for facts.

#Why the hybrid QoS split won

The model is a hybrid: a small number of stable subjects (the payload-event approach, where event_type lives in the payload) split only along the QoS line into .log and .signal (the typed-leaves approach, where the subject leaf encodes a class). The two extremes both lose:

ApproachWhat it doesWhy it loses alone
Pure payload-eventOne subject per interaction; every event type in the payload.Cannot apply different QoS — durable facts and ICE floods share one subject and one storage policy.
Pure typed-leavesA distinct subject leaf per event type (.message, .ice, .typing, ...).Subject explosion and brittle ACLs; clients drift toward the .> wildcard, re-creating the flood.
Hybrid (chosen)Split by QoS into .log / .signal; keep event_type in the payload.Wins: exactly the two QoS classes that matter map to JetStream vs core NATS, while business events stay flexible in the payload.

The QoS line is one boundary that demands a separate subject, because it maps to a hard infrastructure choice — JetStream (durable, ordered) versus core NATS (ephemeral, at-most-once). The second boundary is the authority line: the write-only command subject .cmd versus the router-only .log. That split exists for security (no authorship spoofing) and integrity (single-writer sequence), not QoS — see why the router is authoritative. Everything else is a payload concern, so it stays in the envelope.

#Multi-tenant ACLs

Every subject is tenant-prefixed and protected by per-tenant ACLs. A user authenticated for one tenant cannot read or write another tenant's subjects, and gains interaction access only after accepting the offer.

  • Tenant isolation — a user authenticated for tenant A subscribing to tenant.<B>.interaction.*.> is denied by NATS.
  • Payload tenant match — even on a correctly tenant-prefixed .cmd subject, the router rejects a command whose payload tenant_id does not match the subject's tenant. The subject ACL passing is not enough.
  • ACL after accept — before accepting an interaction's offer, a user subscribing to that interaction.<id>.log is denied; the grant for tenant.<tid>.interaction.<id>.> is issued only on accept.
  • Scoped grants — a user authorized for interaction X holds only tenant.<tid>.interaction.X.>, never interaction.*, so they cannot reach an interaction Y they did not accept.
  • Log is router-authored — clients hold no write ACL on .log; they write only .cmd and their own .signal.<self>. On .cmd the router validates author == the connection's authenticated identity; on .signal.<self> the author is the subject itself, enforced by the publish-ACL.

#Auth callout and reconnect-on-accept

Static permissions cannot express per-interaction grants, so Phase 1 enforces them dynamically through the server's auth callout (authorization { auth_callout { … } }), answered by the router. The callout authenticates each connection's short-lived, nonce/audience-bound, actor+interaction-scoped token and mints that connection's subjects — initially only the user's own routing.offer.user.<self> (+ .control), notify.<self>, presence.<self> (read), and interaction.<accepted>.cmd + .signal.<self> write, .signal.* + .log read for interactions already accepted. It holds NO blanket interaction.* and NO .log write.

Because the callout authorizes at CONNECT time, an already-connected nats.ws client cannot be widened in place. On offer-accept the router issues the client a short-lived interaction-scoped token and the client reconnects (transparent over nats.ws); the callout mints the new connection's subjects, now including ONLY tenant.<tid>.interaction.<id>.> for that one interaction. A user authorized for interaction X cannot touch interaction Y.

Privileged controls (cancel/withdraw/transfer/ACL grant-revoke) are audited with actor and reason on routing.audit.> (the ROUTING_AUDIT stream).

#Token expiry and connection lifetime

The callout runs only at CONNECT, so token expiry mid-session is bounded by an enforced maximum NATS connection lifetime (or a ledger that actively kills connections whose token expired). When the token expires the connection is terminated at the enforced max lifetime; the client refreshes its scoped token and reconnects transparently. This is what keeps a revoked or expired grant from outliving its intended window without a per-message authorization check.

Note Phase 1 auth is user/pass or token at the NATS edge, with per-interaction subjects granted through the auth callout on offer-accept and a max connection lifetime bounding expiry. NKEY/JWT auth is deferred to a later phase.
