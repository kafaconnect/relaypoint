Subject Model

The heart of RelayPoint: the unified-Interaction NATS subject layout, the event envelope, the durable-vs-ephemeral QoS split, the JetStream streams, offer/RONA mechanics, client subscription discipline, the Desk mapping, and multi-tenant ACLs.

#Overview

Every chat, voice, video, email, or social exchange is modeled as one Interaction. The medium is a field in the event payload, never part of the subject — there are no medium-specific subjects such as call.invite or chat.offer. A single conversation that starts as chat and escalates to a voice call keeps the same interaction subjects throughout; only the event_type and medium in the payload differ.

Per interaction there are exactly two update subjects, split by quality of service: a durable, ordered .log on JetStream for facts, and an ephemeral .signal on core NATS for high-rate transient signals. The offer/ring that brings a user into an interaction is a separate, target-addressed routing concern with its own subtree and timeout/RONA semantics — never on the interaction update subjects. All subjects are tenant-prefixed with tenant.<tenantId>. (omitted below for brevity).

Note Subjects are dot-separated and lowercase. Ids (tenantId, interaction id, userId) are ULID/UUID with no dots or slashes, so they never break the subject hierarchy or the a.b.c ↔ a/b/c MQTT auto-map.

#Event envelope

Every message — on .log or .signal — carries the same envelope. The business meaning lives in event_type and the medium lives in medium, both inside the payload.

{
  "schema":      "...",          // schema id / version of `data`
  "event_type":  "message.created",
  "event_id":    "<ULID>",
  "sequence":    42,             // per-interaction ordering
  "occurred_at": "2026-06-08T12:00:00Z",
  "medium":      "chat",         // chat | voice | video | email | social
  "data":        { /* event-specific */ }
}
FieldMeaning
schemaSchema id/version of the data payload.
event_typeThe business event; clients filter on this locally.
event_idUnique id for the event (idempotency / dedup).
sequencePer-interaction ordering counter.
occurred_atEvent timestamp.
mediumThe channel — chat, voice, video, email, social. This is the unified-Interaction key: the medium is data, not a subject.
dataEvent-specific payload.

#QoS split: log vs signal

The split is by quality of service, not by business domain. Durable facts that must be ordered and replayable go on .log (JetStream). High-rate transient signals that must never be stored go on .signal (core NATS). The event_type distinguishes specific events within each subject.

#.log event types

Durable, ordered facts on interaction.<id>.log (JetStream):

  • interaction.started / interaction.ended
  • message.created / message.updated / message.deleted
  • participant.joined / participant.left
  • interaction.accepted / interaction.assigned / interaction.transferred / interaction.held / interaction.resumed
  • webrtc.offer / webrtc.answer
  • call.accepted / call.hold / call.ended
  • recording.started / recording.stopped
Note WebRTC SDP offer/answer are durable state and live on .log — a reconnecting participant must replay them to rebuild the call. ICE candidates are not state and live on .signal.

#.signal event types

Ephemeral, at-most-once signals on interaction.<id>.signal (core NATS, never JetStream):

  • webrtc.ice — ICE candidates
  • typing.* — typing indicators
  • presence.cursor — live cursor position
  • media.level — audio level meters

#JetStream streams

Streams are defined by leaf class, not per interaction. Two streams are required; one is optional; nothing else is stored.

StreamSubjects capturedStorage / notes
INTERACTION_LOGStenant.*.interaction.*.logFile storage; preserves per-interaction ordering.
NOTIFICATIONStenant.*.notify.*Durable; survives reconnect.
ROUTING_AUDIT (optional)tenant.*.routing.audit.>Audit only — NOT live offer delivery.
Danger There is NO JetStream stream for .signal, for presence.*, or for the live offer subjects (routing.offer.*). Capturing .signal in JetStream would persist ICE floods and typing noise and defeat the entire QoS split.

#Offer and RONA

An offer brings a user who is not yet in an interaction into it. The router publishes a NATS request/reply to routing.offer.user.<userId> (team and queue variants exist) carrying a timeout_ms. Offer delivery never shares a subject or stream with interaction updates.

  • Accept within timeout — the client replies accept before the deadline; the router grants the user ACL on interaction.<id>.* and returns the interaction_id.
  • No answer or reject → RONA — if no valid accept arrives before the timeout, or the client replies reject, the router requeues the interaction (redirect-on-no-answer) and the user is not joined.
router --req--> routing.offer.user.<userId>   (timeout_ms)
                       |
          +------------+-------------+
          | accept (in time)         | timeout / reject
          v                          v
   grant ACL on interaction.<id>.*  RONA: requeue interaction
   return interaction_id            user NOT joined

#Subscription discipline

Clients subscribe to specific subjects and filter by event_type locally. They MUST NOT use the interaction.<id>.> wildcard, which would flood the client with every ICE candidate and signal.

  • Subscribe to the exact subjects you need: interaction.<id>.log, interaction.<id>.signal, routing.offer.user.<userId>, notify.<userId>, presence.<userId>.
  • Filter by event_type in the client, not with a subject wildcard.
  • The WebRTC subsystem subscribes to ICE (on .signal) only when the user joins the call — not before — so a chat-only participant never receives ICE traffic.
Warning Never subscribe to interaction.<id>.>. The wildcard collapses the QoS split at the client and re-introduces the ICE flood the .log/.signal separation exists to prevent.

#Why the hybrid QoS split won

The model is a hybrid: a small number of stable subjects (the payload-event approach, where event_type lives in the payload) split only along the QoS line into .log and .signal (the typed-leaves approach, where the subject leaf encodes a class). The two extremes both lose:

ApproachWhat it doesWhy it loses alone
Pure payload-eventOne subject per interaction; every event type in the payload.Cannot apply different QoS — durable facts and ICE floods share one subject and one storage policy.
Pure typed-leavesA distinct subject leaf per event type (.message, .ice, .typing, ...).Subject explosion and brittle ACLs; clients drift toward the .> wildcard, re-creating the flood.
Hybrid (chosen)Split by QoS into .log / .signal; keep event_type in the payload.Wins: exactly the two QoS classes that matter map to JetStream vs core NATS, while business events stay flexible in the payload.

The QoS line is the only boundary that demands a separate subject, because it maps to a hard infrastructure choice — JetStream (durable, ordered) versus core NATS (ephemeral, at-most-once). Everything else is a payload concern, so it stays in the envelope.

#Desk mapping

How the model maps onto KafaConnect Desk's domain:

RelayPointDesk
interactionDesk Session
.log eventsCustomerTimeline facts
.signal eventsNot in the timeline — unless a signal causes a durable state change, in which case it is re-emitted on .log

Because the timeline is built purely from .log, it is fully reconstructable by replaying the INTERACTION_LOGS stream for the session. Ephemeral signals (typing, ICE, cursor, audio level) leave no timeline trace by design.

#Multi-tenant ACLs

Every subject is tenant-prefixed and protected by per-tenant ACLs. A user authenticated for one tenant cannot read or write another tenant's subjects, and gains interaction access only after accepting the offer.

  • Tenant isolation — a user authenticated for tenant A subscribing to tenant.<B>.interaction.*.> is denied by NATS.
  • ACL after accept — before accepting an interaction's offer, a user subscribing to that interaction.<id>.log is denied; the router grants interaction.<id>.* access only on accept.
Note Phase 1 auth is user/pass or token at the NATS edge plus subject ACLs scoped per tenant, with interaction ACLs granted on offer-accept. NKEY/JWT auth is deferred to a later phase.
